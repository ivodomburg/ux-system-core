# Taxonomy — Generiek UX Kennissysteem

## Doel

Deze taxonomy definieert bestandstypen en hun betekenis.

Het doel is:

- consistente interpretatie van bronnen  
- juiste weging van informatie  
- betere AI-analyse en output  
- schaalbaarheid van het systeem  

---

## Structuur

De taxonomy bestaat uit twee niveaus:

- Niveau 1: hoofdcategorieën (stabiel, klein)  
- Niveau 2: subtypes (uitbreidbaar, UX-specifiek)  

---

# Niveau 1 — Hoofdcategorieën

## Project Configuration

Stuurt gedrag, interpretatie en systeemlogica.

Voorbeelden:
- purpose
- instructions
- taxonomy
- transfer_context
- core_contract

Eigenschap:
- hoogste prioriteit bij interpretatie

---

## Architecture

Beschrijft structuur van systemen, flows en informatie.

Voorbeelden:
- core_structure
- knowledge_ontology
- flow definitions
- information architecture
- reference architecture

---

## Design Documentation

Beschrijft UX- en UI-oplossingen.

Voorbeelden:
- wireframes
- component specs
- interaction designs
- UX doctrines
- principles overviews

---

## UX Analysis

Bevat inzichten, onderzoek en evaluaties.

Voorbeelden:
- user research
- usability findings
- analyses
- journey reviews

---

## Reference Context

Bevat externe kennisbronnen die context geven aan het project.

Voorbeelden:
- developer documentation summaries
- platform summaries
- organization summaries

Eigenschap:
- feitelijke referentielaag voor analyse en ontwerp

---

## Project Operations

Bevat organisatorische en besluitvormende projectinformatie.

Voorbeelden:
- meeting notes
- agenda's en spreekpunten
- planning
- goals

Eigenschap:
- stuurt voortgang en besluitvorming, niet directe UX-oplossing

---

## Methods

Definieert acties en workflows.

Voorbeelden:
- method_registry
- method definitions

# Interpretatieregels

## Prioriteit

1. Project Configuration  
2. Methods  
3. Architecture  
4. Design Documentation  
5. Reference Context  
6. UX Analysis  
7. Project Operations  

Configuratie heeft altijd voorrang.

---

## Gebruik

- Elk bestand krijgt minimaal één type  
- Subtypes zijn optioneel maar aanbevolen  
- Interpretatie gebeurt op basis van type + context  
- Foldercontext mag gebruikt worden als aanvullende classificatiehint  

---

## Regels

- Types mogen niet overlappen in betekenis  
- Subtypes moeten logisch onder hoofdcategorie vallen  
- Nieuwe types worden expliciet toegevoegd  
- Geen projectspecifieke types in core  
- Overlay mag type-labels gebruiken die mappen op gedefinieerde subtypes  

---

## Uitbreiding

Uitbreiding van de taxonomy gebeurt via:

- toevoegen van nieuwe subtypes  
- niet via wijzigen van hoofdcategorieën  

Nieuwe types moeten:

- generiek toepasbaar zijn  
- duidelijk onderscheid hebben  
- niet overlappen met bestaande types  

---

## Doel van deze taxonomy

Zorgen dat:

- AI bestanden correct interpreteert  
- analyses consistenter worden  
- systeem schaalbaar blijft  
- hergebruik mogelijk blijft  
- overlay-bestanden formeel valideerbaar blijven tegen de core
